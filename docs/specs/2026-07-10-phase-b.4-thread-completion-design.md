# Phase B.4: Thread 完整化 — 设计

> **状态**:设计(2026-07-10)。承接 B.3a/b/c(阻塞管程 + start0 + Object.wait/notify/notifyAll)。
> **目标**:让真 Java 多线程程序端到端跑通:`new Thread(r, name).start(); t.join()` + 中断语义。
> **北极星**:退役"start0 直调 + Rust join"侧门,使真 `Thread` API 在 Java 层闭环。

---

## 1. 背景:B.3 之后还差什么

B.3a/b/c 把**原语**层做完:

| 层 | 成果 | 闸门侧门 |
|---|---|---|
| B.3a | 真 `JavaMonitor`(Condvar 阻塞) | std::thread 直测管程 |
| B.3b | `start0` 真起 OS 线程 + eetop 生命周期 + JoinHandle 表 | **`new_instance` 不跑 `<init>`**(绕过构造器)+ Rust `join_thread` |
| B.3c | `Object.wait/notify/notifyAll` 真阻塞 | 单线程 timeout/no-op/IMSE 闸 |

即:**线程能起、能 wait/notify,但 Java 层 `new Thread()` 构造器从未跑过**(B.3b 闸用 `new_instance` 旁路 `<init>`)。B.4 把这条缝补上,并补 `join()` 与中断。

## 2. 源码核查(Step 0,§3)

- `Thread.start()`(Thread.java:1465):`synchronized(this){ if(holder.threadStatus!=0) throw IMSE; start0(); }` —— 读 `holder.threadStatus`,调已实现的 `start0`。
- `Thread.run()`(1523):`Runnable task=holder.task; if(task!=null) runWith(bindings,task);` —— 基类 `run` 从 `holder.task` 取 Runnable 调其 `run()`(B.3b 的虚分派已覆盖 override 场景;基类路径需 `holder.task` 可读)。
- `Thread.join(long)`(1901):`synchronized(this){ while(isAlive()) wait(0); }`,`isAlive()`(1713)= `eetop!=0`。**终止时需 `notifyAll(this)` 唤醒 joiner**(HotSpot `JavaThread::exit`→`ensure_join`:`ObjectSynchronizer::enter`→`notifyall`→`exit`)。
- 构造器(739 `Thread(ThreadGroup,String,int,Runnable,long)`):
  - `parent=currentThread()`(✅),`parent==this?`(新线程→false)
  - `g==null → g=parent.getThreadGroup()` → **需 main 线程有 holder.group**
  - `priority=Math.min(parent.getPriority(), g.getMaxPriority())` → 需 holder.priority / ThreadGroup.maxPriority
  - `new FieldHolder(g,task,stackSize,priority,parent.isDaemon())` → 需 `Thread$FieldHolder` 可实例化
  - `tid=ThreadIdentifiers.next()` → `Unsafe.getAndAddLong(null,NEXT_TID_OFFSET,1)`
  - `name=(name!=null)?name:genThreadName()` → **null 名走 `genThreadName()`**
  - `scopedValueBindings=NEW_THREAD_BINDINGS`(静态 Object,无依赖)
- `genThreadName()`(1120)= `"Thread-"+ThreadNumbering.next()`;`ThreadNumbering.<clinit>` 用
  `U.staticFieldBase/Offset(Field)` —— **反射原生(4.15b,阻塞)**。→ null 名构造器暂不可达。
- `ThreadIdentifiers.<clinit>`:`NEXT_TID_OFFSET=Thread.getNextThreadIdOffset()`(native,需绑,返静态
  `nextThreadId` 字段 ord);`ThreadIdentifiers.next()`=`getAndAddLong`=字节码
  `do{v=getLongVolatile(o,off)}while(!weakCompareAndSetLong(...))` —— **需绑 `getLongVolatile`(o=null 静态)**;
  `weakCompareAndSetLong`→`compareAndSetLong`(已绑,须确认 o=null 静态分支)。
- 系统 `ThreadGroup()` 私有构造器(ThreadGroup.java:102):`parent=null;name="system";maxPriority=Thread.MAX_PRIORITY(10)`,3 字段赋值。`getMaxPriority()`(175)读 `maxPriority` 字段。
- `Thread$FieldHolder`(Thread.java:281):`group/task/stackSize/priority/daemon/threadStatus/nativeThreadID/terminatingThreadLocals`。"All fields except task and terminatingThreadLocals are accessed directly by the VM"。

## 3. 关键决策

### D1:null 名构造器顺延(反射墙)
`new Thread(r)`(null 名)→ `genThreadName` → `ThreadNumbering.<clinit>` →
`Unsafe.staticFieldBase/Offset` → 真 `Field` 反射(Layer 4.15b,待做、阻塞于 MethodHandle)。
**B.4 闸统一用显式名 `new Thread(r, "w")`**(走 `Thread(Runnable,String)`,name 非 null → 绕开 genThreadName),
忠实 null 名路径顺延 4.15b。这与既有"顺延"项(精确 float 串化等)同性质。

### D2:main 线程补 holder(VM 引导,非跑 <init>)
HotSpot primordial 线程的 group/holder 由 `Threads::create_vm` 置(非跑 Java <init>,存在 chicken-and-egg)。
rustj 同理:`alloc_main_thread` 当前仅置 `name/tid`;B.4a **额外置 `holder` 字段**——分配 `Thread$FieldHolder`
实例 + 系统 `ThreadGroup` 实例,置 holder.{group=系统组, priority=NORM_PRIORITY(5), daemon=false, task=null,
stackSize=0, threadStatus=0}。`new_instance` 不跑 <init>,按名写字段(同 main_thread 既有模式)。

### D3:`Thread$FieldHolder` / 系统 `ThreadGroup` 经 load_closure 载真类
闸门需先 `load_closure` 载 `java/lang/Thread`(传递性载 `Thread$FieldHolder` 嵌套类)+ `java/lang/ThreadGroup`。
`new_instance` 真 FieldHolder/ThreadGroup,按名置字段。系统组名="system"、maxPriority=10。

### D4:`getLongVolatile` + `getNextThreadIdOffset` 原生
- `Thread.getNextThreadIdOffset()J`(private static native,Thread.java:2628):返静态 `nextThreadId` 字段 ord(i64)。
  按 §6 注册表 ord 查询(静态字段区)。初始值 0(默认)。
- `Unsafe.getLongVolatile(Object,long)J`:**o=null → 静态 long 字段按 ord 读**;o=Instance → 实例 long 字段按 ord 读
  (镜像既有 volatile 读的 ord 模型,单线程无屏障)。**非数组**(既有 `getLong` 仅 byte[];本原生走实例/静态,不复用)。
- `compareAndSetLong(o,off,exp,new)` 已绑;**确认 o=null 静态分支**(无则补)。

## 4. 分层(沿用 B.3 a/b/c 节奏)

| 子层 | 内容 | 闸门 |

## 5. B.4a 任务分解(TDD)

1. **RED**:lib 单测 `thread_constructor` —— 建 registry(jmod 载 Thread/ThreadGroup + 显式名构造的 Probe 类)、
   调 `new Thread(runnable,"w")`、断 `getName()=="w"`、`getId()>0`、`getThreadGroup()` 非 null 且 `getName()=="main"`(或系统组)、`getPriority()==5`、`isDaemon()==false`。先证构造器抛/失败(缺 holder→NPE,或缺 native→ULE)。
2. **bind `Thread.getNextThreadIdOffset`** —— 返静态 `nextThreadId` ord。
3. **bind `Unsafe.getLongVolatile`**(实例/静态按 ord 读 long)。
4. **确认/补 `compareAndSetLong` 静态分支**。
5. **main 线程 holder 引导**(alloc_main_thread 增 FieldHolder + 系统 ThreadGroup;惰性,经 load_closure 真类)。
6. **GREEN**:构造器跑通,断言全过。
7. **javac 闸门** `tests/thread_constructor.rs`(同 B.3b 风格:javac 编译 Probe,jmod 载,真字节码跑构造器+字段读)。
8. **commit + memory**。
