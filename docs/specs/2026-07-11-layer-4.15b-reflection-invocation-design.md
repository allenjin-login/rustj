# Layer 4.15b — 反射调用(Method.invoke / Constructor.newInstance)设计

> **状态**:设计(2026-07-11)。`/goal` 主线:用户铁令「多线程完成后记得完成 4.15b」。
> Phase B.4(多线程)已完成(B.4a/b/c + 收尾 `7de7ddd`),本层为反射调用收尾。

## 1. 目标

让真 Java 反射调用端到端跑通:

```java
Method m = Integer.class.getDeclaredMethod("parseInt", String.class);
int v = (int) m.invoke(null, "42");                 // → 42

Constructor<?> c = MyClass.class.getConstructor(int.class);
MyClass obj = (MyClass) c.newInstance(7);           // → new MyClass(7)
```

经**真 java.base 字节码路径**(`Method.invoke` → `MethodAccessor` → native `invoke0`),
非解释器层 short-circuit(北极星:跑真 java.base)。

## 2. Step 0 源码核验(决定性发现)

### 2.1 JDK25 反射调用架构(`jdk-master` + 本机 jmod 一致)

- **`Method.invoke`(Method.java:561)非 native**——字节码:检 access → 取 `methodAccessor`
  (惰性 `acquireMethodAccessor`)→ `ma.invoke(obj, args)`。
- `acquireMethodAccessor` → `ReflectionFactory.newMethodAccessor` →
  **`MethodHandleAccessorFactory.newMethodAccessor`**(jdk/internal/reflect/)。
- **`useNativeAccessor(member)`(MethodHandleAccessorFactory.java:360)首条**:
  `if (!VM.isJavaLangInvokeInited()) return true;`。
- **`VM.isJavaLangInvokeInited()`(VM.java:104)读 `@Stable boolean javaLangInvokeInited`(默认 false)**;
  仅 `setJavaLangInvokeInited()`(System.initPhase3)置 true。**rustj 不跑 initPhase3 → 恒 false** →
  `useNativeAccessor` 恒 true → **NativeAccessor 路径**(非 MethodHandle 路径)。
- ⇒ **绕过「MethodHandle 直接调用」墙**(原 4.15b 顺延阻塞点):NativeAccessor 走 **VM native 反射**。

### 2.2 待 bind 的 native(本机 jmod `javap` 实测,描述符以 jmod 为准)

```
jdk/internal/reflect/DirectMethodHandleAccessor$NativeAccessor
    private static native Object invoke0(Method m, Object obj, Object[] args);
    // (Ljava/lang/reflect/Method;Ljava/lang/Object;[Ljava/lang/Object;)Ljava/lang/Object;
    = HotSpot JVM_InvokeMethod (jvm.cpp)

jdk/internal/reflect/DirectConstructorHandleAccessor$NativeAccessor
    private static native Object newInstance0(Constructor<?> c, Object[] args)
        throws InstantiationException, InvocationTargetException;
    // (Ljava/lang/reflect/Constructor;[Ljava/lang/Object;)Ljava/lang/Object;
    = HotSpot JVM_NewInstance (jvm.cpp)
```

两者 `private static native` → 经 `native::invoke` 分派表 bind(同 4.10c 模式)。
`DirectMethodHandleAccessor$NativeAccessor.invoke`(字节码)断言 `csmAdapter==null` 后直调 `invoke0`;
`DirectConstructorHandleAccessor$NativeAccessor.newInstance`(字节码)直调 `newInstance0`。

### 2.3 Field.get/set —— **本层顺延**

- 本机 jmod **无 NativeFieldAccessor**(仅 `MethodHandle*FieldAccessorImpl`,9 个按类型分)。
- `MethodHandleAccessorFactory.newFieldAccessor`(行 172)`if (!VM.isJavaLangInvokeInited())`
  **抛 InternalError**——即 Field 反射**强制要求 java.lang.invoke 已初始化**(MethodHandle 路径)。
- ⇒ rustj 无 MethodHandle 直接调用前,**Field.get/set 不可达**。顺延至 MethodHandle 层(候选 g)。

### 2.4 slot 解析(`Method.clazz` + `Method.slot` → 目标 MethodInfo)

- 4.15a `getDeclaredMethods0(false)`(java_lang.rs:934)在 `publicOnly=false` 时**无过滤**
  → `enumerate()` 索引 == `cf.methods` 原始索引 → **`slot` = `cf.methods` 原始下标**。
- 公开 API `Class.getDeclaredMethod` → `privateGetDeclaredMethods(false)` → `getDeclaredMethods0(false)`
  → slot 保留(JDK `searchMethods`/`copyMethod` 保 clazz+slot)。✓
- ⇒ **invoke0/newInstance0 据 `Method.clazz`(Class 镜像→内部名)+ `Method.slot`(i32)直接索引
  `cf.methods[slot]`** 解析目标(HotSpot 同语义)。`getMethod`(publicOnly=true 路径)slot 可能错位
  → 当前仅保证 `getDeclaredMethod` 路径;.getMethod 顺延(slot 写器改写为原始索引,可选后续)。

## 3. 架构

### 3.1 `invoke0(method, obj, args)` → JVM_InvokeMethod 语义

1. 读 `Method.clazz`(引用字段→Class 镜像→`mirror_internal_name`)、`Method.slot`(int 字段)、
   `Method.modifiers`(int 字段)。
2. 取 clazz 的 `LoadedClass`,`cf.methods[slot]` = 目标 MethodInfo(读 name + descriptor)。
3. **ACC_STATIC**(`modifiers & 0x0008`):
   - 静态:目标即 `cf.methods[slot]`;`locals` = unbox(args) 按 descriptor 形参类型。
   - 实例:**虚分派**——读 `obj` 运行时类 → `resolve_dispatch(runtime_class, name, desc)`(子类
     override 优先);`locals[0] = obj`,`locals[1..]` = unbox(args)。`obj==null` + 实例法 → NPE。
4. 建 `Frame(max_locals, max_stack)` → `interpret_with`。
5. **返回装箱**:据 descriptor 返回类型(`V`→`null`/void;原语 `I/Z/B/C/S`→`Int`、`J`→`Long`、
   `F`→`Float`、`D`→`Double` 装箱为对应包装类实例;引用→原 `Reference`)。
6. **异常包装**:`interpret_with` 返 `ThrownException(t)` → 抛 `InvocationTargetException` 并置其
   `cause` 字段 = t(`Method.invoke` 契约:目标方法抛的异常包 ITE)。`InvocationTargetException`
   已在 java.base(bootstrap 已含)。

### 3.2 `newInstance0(ctor, args)` → JVM_NewInstance 语义

1. 读 `Constructor.clazz`、`Constructor.slot`。
2. `new_instance`(不跑 `<init>`)分配 clazz 实例。
3. `cf.methods[slot]`(须为 `<init>`)为目标;`locals[0] = 新实例`,`locals[1..]` = unbox(args)。
4. `interpret_with` 跑 `<init>`(无返回值)。
5. 返新实例;`ThrownException` → 包 ITE(同上)。抽象类/`Class`实例化 → InstantiationException
   (JDK 在 `newConstructorAccessor` 前置检,rustj 在 native 内同样检 `ACC_ABSTRACT`/`clazz==Class`)。

### 3.3 参数 unboxing(`Object[] args` → `Slot`s,按 descriptor 形参)

- 形参 `I/Z/B/C/S`:读包装实例 `value` 字段 → `Slot::Int`(`Z`→0/1,`C`→utf16 码点)。
- `J`:`Long value` → `Slot::Long`。
- `F`:`Float value` → `Slot::Float`。
- `D`:`Double value` → `Slot::Double`。
- 引用类型:数组元素 `Reference` 直用(`null` 保留)。
- `args==null` 视为零长;长度 ≠ 形参数 → `IllegalArgumentException`(NativeAccessor.invoke 字节码
  本先 `checkArgumentCount`,但 native 兜底)。

### 3.4 返回装箱复用

复用既有装箱通路(4.10ac `stringify_arg` / Integer/Boolean/… 镜像分配模式)。原语返回须装箱为
包装类实例(`box_primitive(vm, ty, slot) -> Reference`),引用/`V` 直返。

## 4. TDD 计划(probe-driven,同 4.17-4.40 方法论)

真 `Method.invoke` 字节码路径可能顺带要求中间 native(`Reflection.isCallerSensitive` 字节码经
注解机制 → 可能触发 `getDeclaredAnnotations0` 等)。**逐个 RED→bind→GREEN**,不预算。

### Gate 1:`tests/reflection_invoke.rs`(Method.invoke)

```java
public class Probe {
    public static int invokeParseInt() throws Exception {
        Method m = Integer.class.getDeclaredMethod("parseInt", String.class);
        return (int) m.invoke(null, "42");              // 期望 42
    }
    public static int invokeInstance() throws Exception {
        Method m = String.class.getDeclaredMethod("length");
        return (int) m.invoke("hello", new Object[0]);  // 期望 5
    }
    public static int invokeOverload() throws Exception {
        Method m = Integer.class.getDeclaredMethod("parseInt", String.class, int.class);
        return (int) m.invoke(null, "ff", 16);           // 期望 255
    }
}
```

- **RED**:首缺(预期 invoke0 ULE;或中间 native ULE)。
- **GREEN**:bind `invoke0`(jvm_InvokeMethod 移植)+ 中间缺失 native,三法绿。

### Gate 2:`tests/reflection_newinstance.rs`(Constructor.newInstance)

```java
public class Probe {
    public int v;
    public Probe(int x) { v = x; }
    public static int newInstance() throws Exception {
        Constructor<?> c = Probe.class.getConstructor(int.class);
        return ((Probe) c.newInstance(7)).v;            // 期望 7
    }
}
```

- **RED/GREEN**:bind `newInstance0`,绿。

### Gate 3(lib 单测):slot/unbox/box 纯单元

- `invoke0` 解析 `Integer.parseInt(String)` slot → 调用返 42(钉 slot 解析 + unbox + Int 装箱)。

  - bind `DirectConstructorHandleAccessor$NativeAccessor.newInstance0` → `new_instance_reflective`。
  - 路由(新 `jdk/internal/reflect/DirectMethodHandleAccessor$NativeAccessor` + `…Constructor…`)。

  registry 侧)、`unbox_args` / `box_primitive`(新,反射调用专用,或放 vm 模块)。
- **bootstrap**:`java/lang/reflect/InvocationTargetException` 加入 BOOTSTRAP_HIERARCHY
  (Exception 子类;ITE.cause 字段经 `Throwable.target` 或自有 cause——核验 jmod 实测字段名)。

## 6. 顺延项(已知,非本层)

- **Field.get/set**:阻塞于 MethodHandle 直接调用(候选 g)。jmod 无 NativeFieldAccessor。
- **`getMethod`(publicOnly=true)slot 错位**:可选后续改 `getDeclaredMethods0` 写器(slot=原始索引)。
- **caller-sensitive 方法反射调用**:`NativeAccessor.invoke(obj, args, caller)` 三参重载 + CS 适配法;
  当前仅绑二参 `invoke0`(非 CS 路径)。CS 方法反射顺延。
- **注解机制**:若 `Reflection.isCallerSensitive` 字节码路径触发 `getDeclaredAnnotations0`,顺带 bind。
